# Systems Philosophy

## What "Systems" Means (Beyond Literal Automation)

**Systems** here is not only about software automation. It's about **reducing cognitive load** by externalizing decisions and processes so you:

- **Don't repeat** the same planning each time (templates, reusable checklists)
- **Don't forget** (move tasks and details out of your head into a system)
- **Stay productive** (allocate attention and energy instead of grinding more)

## Dr. Justin Sung: Three Principles for Thinking in Systems

*Source: [How to Build Systems to Actually Achieve Your Goals](https://www.youtube.com/watch?v=fXyRprdoEoE) – 2.3M views*

**Core idea**: Reduce reliance on willpower and motivation. Build processes that automatically help achieve results. Chain processes together = system.

| Principle | Description |
|-----------|-------------|
| **1. Think holistically** | List all factors that could cause the plan to fail. Expect obstacles (tired, busy, something comes up). Ask: "What can I do about that?" |
| **2. Build for repeatability** | System must work on the **worst day**. Minimize friction. Cycle: identify high-effort step → find ways to reduce it → check what might still break it → repeat until you find a working combination. |
| **3. Peel the band-aids** | First version uses short-term fixes (e.g. naps, timers). Address root causes (sleep, focus) over time so the system doesn't depend on fragile conditions. |

**Outcome**: Plans become explicit about contingencies ("If tired, do X. If not tired, do Y."). System adapts to reality instead of relying on rigid willpower.

## Principles for Systems App

1. **Externalize, don't memorize** – Packing lists, trip prep, people notes live in the app, not in your head.
2. **Templates over blank slate** – 80% pre-filled; copy from previous or from community.
3. **Reduce decision fatigue** – Defaults and structures so you decide less often.
4. **Design for endurance** – Works when energy is low, not only in peak focus (aligns with repeatability).
5. **Adapt to reality** – System bends for unplanned events; avoid rigidity.
6. **Contingencies in the system** – Plans should encode "if X, then Y" so the app supports dynamic behavior, not rigid routines.
